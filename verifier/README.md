# Bundle Verifier (static, client-side)

A single self-contained HTML page that verifies the integrity of a
`verifiable-memory-bundle` export — entirely in the browser. No backend, no
build step, no external dependencies. Once loaded, it works offline. Nothing
is sent to any server.

## What it checks

For every entry, in chain order (exact replica of `src/tools/chain.ts`):

1. `contentHash` — recomputed SHA-256 of `content`.
2. `entryHash` — recomputed SHA-256 of the canonical
   `JSON.stringify({contentHash, prevHash, createdAt})` (key order matters).
3. `prevHash` — must equal the previous entry's `entryHash`.

Optionally, it compares the SHA-256 of the **exact file bytes** against an
externally published anchor hash (input field or `?anchor=` query param).
This is what detects a full rewrite of an internally-consistent bundle.

Verdicts: **INTACT** (green) / **ALTERED** (red). Unreadable files or a
malformed anchor produce a neutral error instead of a false verdict — unless
a valid anchor is provided, in which case the anchor comparison decides.

## Usage

- Serve `index.html` over `https://` or `http://localhost` (WebCrypto
  requires a secure context), or open it from any static host.
- Load a bundle via file picker, drag & drop, or `?bundle=<url>`.
- Optional: paste the anchor hash, or pass `?anchor=<sha256hex>`.
- UI is bilingual (ES/EN toggle).

Generate a bundle with the `export` tool of the MCP server; anchor it by
publishing the file's SHA-256 somewhere you don't control.

## Acceptance evidence

`node verifier/acceptance.mjs` spins up the real MCP server against a
temporary database, exports a real bundle, extracts the **exact** core
script from `index.html`, and runs it through 9 scenarios (intact, edited
content, reordered entries, deleted entry, anchor mismatch, truncation,
invalid inputs). It exits non-zero on any failure.

`sample-bundle.json` + `sample-bundle.sha256` are a demo artifact pair
generated by that script (simulated content).

## Honest scope

Tamper-evident, not tamper-proof: this verifier detects alteration of the
record; it does not validate that the content itself is true or correct.
